# 群购订单管理系统 (Group Order Management H5)

> 给 Cursor / AI 助手：请先阅读本文件，再阅读 `docs/` 下各篇；**若需确认「需求与当前代码是否一致」**，优先读 **`docs/11-需求与实现对照.md`**，并读 `HANDOVER.md`（环境/入口约定）。**同一订单内支付组**边界与统计口径见 **`docs/CONSTITUTION_支付组.md`**（与 `docs/04` §4.6 互补）。理解整体后再开始编码。

---

## 一句话定位

**为马来西亚群购/外卖商户设计的 H5 订单管理系统。**

商户在微信群、WhatsApp 群里发布订餐链接，顾客点击链接下单、上传付款截图，商户在后台核对收款、管理出餐。

---

## 核心场景

**典型用户：** 一位在小区群里做家庭外卖的店主"辉姐"
- 每天中午发布"今日晚餐"链接到几个微信群和 WhatsApp 群
- 顾客点链接选菜下单、扫码付款、上传付款截图
- 辉姐在手机上核对每笔订单的付款，按配送点配餐
- 5 点准时截止接单，开始做饭，6 点统一送餐到几个固定地点

---

## 替代什么产品

替代群报数小程序（qun100.com）。群报数的痛点：
1. **绑定微信生态** — WhatsApp、Telegram 用户用不了
2. **顾客端有第三方广告** — 污染商户品牌
3. **分享卡片显示群报数 Logo** — 抢商户品牌
4. **数据统计不分有效订单** — 已付款和未付款混在一起，对账麻烦
5. **缺少配送点系统** — 商户要自己看地址归类
6. **付款核对靠手工** — 翻聊天记录找截图
7. **不专门为外卖/团购设计** — 是通用工具，每个场景都用得别扭

---

## 我们的差异化

1. **跨平台 H5** — WhatsApp / Telegram / Line / 微信都能用
2. **顾客端零广告** — 全程商户品牌
3. **分享卡片只显示商户品牌** — 不抢商户流量
4. **三色付款核实** — 🟢 正常 / 🟡 注意 / 🔴 存疑（基于上传时间和 MD5 比对）
5. **配送点系统** — 顾客填地址自动匹配，商户按配送点分组出餐
6. **早鸟活动价** — 商品可设带时间的活动价，到期自动恢复原价
7. **专为群购/外卖场景设计** — 不通用，但每个细节贴合场景

---

## 技术栈

- **前端：** React + Vite + Tailwind CSS
- **后端：** Firebase
  - Firestore（数据库，实时同步）
  - Storage（图片存储）
  - Authentication（手机号验证码）
- **部署：** Vercel
- **商户端：** PWA（可添加到桌面，支持推送通知）
- **顾客端：** 纯 H5（无需安装）

## 给 Cursor 的开发建议

1. **先搭骨架，再做细节** — 先把所有页面的路由和空白页搭起来，再逐个完善
2. **顾客端优先** — 顾客端是核心体验，先把它做好
3. **数据 mock 在前，Firebase 接入在后** — 先用假数据跑通界面，再接入 Firebase
4. **移动端优先** — 大部分用户在手机上访问，PC 是附加场景
5. **简洁优于功能丰富** — 我们已经砍掉了很多边缘功能，保持克制
6. **遇到不确定的设计** — 优先看 `docs/10-设计原则.md`，再问用户

---

## 常见误区警示

❌ **不要做这些（看似合理但已被否决）：**

1. 不要做"取餐二维码"功能（订单编号叫号就够）
2. 不要在顾客端塞任何第三方广告
3. 不要在分享卡片上显示平台 Logo
4. 不要把付款截图功能做成自动判断（必须保留人工确认）
5. 不要做评分、点赞、评论等社交功能
6. 不要默认要求顾客注册（游客可下单是核心）
7. 不要把数据统计混在一起（已付款/待核实/未付款必须分开）
