# 16 - 饭团配送员

> **文档性质**：饭团配送员（Courier）功能的产品与实现规格；**仅适用于饭团渠道**，商户自营店铺 `DeliveryPoints` 不在本模块范围。  
> **实现状态**：截至文档编写时 **未实现**；路由与集合以本文为准，落地后同步 **`docs/11`**、**`docs/06`**、**`HANDOVER.md`**。  
> **关联**：支付组口径见 **`docs/CONSTITUTION_支付组.md`**；配送档见 **`docs/CONSTITUTION_长期项目配送.md`**；饭团配送区/点主数据见 **`web/src/lib/feituanDeliveryService.ts`**、**`/admin/feituan/delivery`**。

---

## 一、产品定位

饭团运营需要把「已接单、待配送」的订单按**配送档 → 配送区 → 配送点**交给一线配送员执行。本模块提供：

1. **饭团后台**：维护配送区/点（已有）+ **配送员账号与负责点范围**（新增）。
2. **配送员 H5**：手机号 + 密码登录，查看**当前进行中配送档**的任务，到点拍照并完成配送；支持反选异常单。
3. **与顾客/商户登录分离**：不走 Firebase 顾客手机验证码链路，不走饭团管理员 `feituan_admins` 入口。

---

## 二、范围边界

| 在范围内 | 不在范围内 |
|----------|------------|
| 饭团订单（`channel === 'feituan'`） | 商户自营 `/shop/...` 配送点 |
| `feituan_delivery_sets` 区/点 | 商户 `delivery_points` 店铺库 |
| 按 `deliverySlot` 分档任务 | 用 `closesAt` 代替配送档 |
| 配送员到点完成 + 异常勾选 + 补拍 | 商户核实收款、对账导出（仍走现有后台） |

## 三、饭团后台：饭团配送（两个功能区）

路径建议：`/admin/feituan/delivery`（Tab 或子路由）。

### 3.1 Tab 1 · 配送区 / 配送点

- **现状**：`FeituanDelivery.tsx` + `feituan_delivery_sets`。
- **行为**：维护配送区名称、描述、启用；区内配送点（简称、编码、地址、地图、图片等）。
- **项目绑定**：仍在 `project-delivery` 配置项目可用哪些区（不变）。

配送点 ID 与订单一致：`deliveryPointId = feituan:{zoneId}:{pointId}`（见 `listActiveFeituanDeliveryPoints`）。

### 3.2 Tab 2 · 配送员管理

| 字段 | 说明 |
|------|------|
| `displayName` | 备注名，运营识别 |
| `phone` | 登录名（建议规范为 E.164 或统一国家码录入） |
| `password` | 仅创建/重置时由管理员录入；库存 **hash**，不明文 |
| `isActive` | 停用后不可登录 |
| `assignedPointKeys` | `string[]`，每项为 `feituan:{zoneId}:{pointId}`；**可跨区多选** |
| `uid` / `courierId` | 文档主键；若用 Custom Token 可关联 Firebase `uid` |

**运营**：创建配送员 → 勾选负责点 → 将初始密码告知配送员。密码策略 **从简**（无最短长度、无首次改密、无重置后强制改密）；实现层仍须 hash + HTTPS + 登录限速。

## 四、配送员端：路由与页面

建议前缀：`/delivery`（与 `/admin/feituan`、`/feituan` 分离）。

| 路径 | 说明 |
|------|------|
| `/delivery/login` | 手机号 + 密码 |
| `/delivery` | 首页：当前配送档 + 切换其他档（只读） |
| `/delivery/zone/:zoneId` | 本档本区：商户来源汇总 + 各点单数 |
| `/delivery/point/:pointKey` | 本档本点：标贴列表、异常勾选、拍照、完成本点 |

`pointKey` = URL 编码后的 `feituan:{zoneId}:{pointId}`。

### 4.1 配送档选择与权限

| 档类型 | 查看 | 作业（完成、异常、新拍照完成） |
|--------|------|--------------------------------|
| **进行中档**（系统默认） | ✅ | ✅ |
| **其他档**（过去/未来） | ✅ 只读 | ❌；允许 **补拍**（见 §7.3） |

- 打开 App **默认**进入「进行中」配送档。
- 配送员可切换查看其他档，但仅 **进行中档** 开放工单操作。

**进行中档判定**（实现时与 `recurringDeliverySchedule`、`deliverySlot` 对齐）：当前时刻落在该档配送执行相关窗口内；具体算法以实现文件 `deliverySlotActive.ts`（待建）为准，并与对账页 `deliverySlotKey` 一致。

## 五、任务口径：按支付组筛选，标贴一单一张

### 5.1 订单是否进入「本档、本点」任务

对订单执行 `buildPaymentGroups` 后：

| 支付组状态 | 计入任务 |
|------------|----------|
| **已确认** | ✅ |
| **待确认**（已发生支付动作） | ✅ |
| **待付款** | ❌ |

订单级额外条件（全部满足才进入该点任务池）：

- `channel === 'feituan'`
- `deliverySlot` 与当前所选配送档一致（`orderMatchesDeliverySlotKey` 同类规则）
- `deliveryPointId` 为标准饭团点，且 ∈ 配送员 `assignedPointKeys`
- 非取消：`status !== 'cancelled'`
- **其他地址 / 未匹配点**：`isManualMatch === true` 或无法解析为饭团标准点 → **不进任何配送员任务**
- 至少有一个支付组满足上表（confirmed / pending），且该组行属于本档配送范围

**不依赖**整单 `status` 字面量（如 `partial_paid`）；以组为准。

### 5.2 标贴（一单一张）

每个进入任务池的订单在该点生成 **1 张标贴**：

| 标贴字段 | 来源 |
|----------|------|
| 配送区 | 区名（`zoneName` / 由 `pointKey` 解析） |
| 配送点 | 点简称 |
| 单号 | `orderNumber` |
| 客户名 | `customerName` |
| 订单简要明细 | 合并本档、本点下 **已确认 + 待确认** 各组对应 `lines`（品名 × 数量，可截断） |

**标贴上不出现**：金额、付款方式、截图、内部备注、「待核实」字样（待确认单与已确认单标贴样式一致）。

### 5.3 区层级汇总（跨项目合并）

进入某区、某档时展示：

**本配送档订单来源**（按商户/店铺名快照统计 **订单数**）：

> 甲商户 12 单、乙商户 8 单、丙商户 3 单，**合计 23 单**

**本区各配送点**（仅列出该配送员负责的点）：

> 1. A 大堂 — 5 单  
> 2. B 入口 — 3 单  

多项目同日配送 **合并** 在同一区/点清单中，不按项目分 Tab。

## 六、点完成：默认全正常，反选异常

完成粒度：**配送点 + 配送档**（不是逐单必须点「已送」）。

### 6.1 交互

1. 配送员进入某点（进行中档）。
2. **无缺失**：无需打开明细 → 可直接「本点已完成配送」+ 可选上传现场照（0 张亦可完成）。
3. **有缺失**：打开明细 → **勾选异常订单**（按订单勾选；未勾选 = 正常送达）→ 可选拍照 → 「本点已完成配送」。

### 6.2 完成记录（建议集合）

`feituan_delivery_point_completions`（命名可实现时微调）：

| 字段 | 说明 |
|------|------|
| `deliverySlotKey` | 与对账/清单一致 |
| `pointKey` | `feituan:{zoneId}:{pointId}` |
| `courierId` | 配送员文档 id |
| `completedAt` | 完成时间 |
| `photoUrls` | 点级现场照（可多张，Storage） |
| `exceptionOrderIds` | 勾选的异常单；**空 = 声明本点本档全部正常** |
| `totalOrderCount` | 完成时任务单数快照（可选） |

只读档：**不可**新建完成、**不可**改异常列表；**可**对已完成记录 **追加** `photoUrls`（补拍）。

## 七、认证与安全（讨论定稿）

### 7.1 登录

- **电话 + 密码**，独立配送员会话。
- **不**使用顾客端 `AccountCenter` 手机验证码流程。
- **建议实现**：Cloud Function `courierLogin` → 校验 `feituan_couriers` 中 hash → 签发 **Firebase Custom Token**（`claims.courierId`）或等价 session；Firestore 规则禁止配送员读全表 `orders`。

### 7.2 权限

| 角色 | 权限 |
|------|------|
| `feituan_admins` / `platform_admins` | 区/点 CRUD、配送员 CRUD、查看完成记录（可选） |
| 配送员 | 仅拉取本人负责点 + 当前档任务；写本点完成与照片 |
| 商户/顾客 | 无配送员入口 |

### 7.3 任务数据拉取

- 不宜客户端 `getDocs(orders)` 全表过滤。
- **建议**：HTTPS Callable / Function `listCourierTasks({ deliverySlotKey })` 服务端按 `deliveryPointId in assignedPointKeys` 分片查询并组 manifest。
- 复用逻辑：`buildPaymentGroups`、`orderMatchesDeliverySlotKey`、区/点聚合思想对齐 `feituanDeliveryReconciliation.buildDeliveryManifest`。

## 九、实现分期（推荐，勿一次合入）

> **建议分阶段交付**，原因：独立认证、新集合、服务端任务 API、Storage 上传、与支付组/配送档耦合多；一次做完难测、难回滚。

| 阶段 | 交付 | 可验收 |
|------|------|--------|
| **P0 · 基础** | 后台 Tab2 配送员 CRUD（点勾选）；`feituan_couriers` + 密码 hash；`courierLogin`；`/delivery/login` | 管理员建账号、配送员能登录 |
| **P1 · 任务只读** | `listCourierTasks`；进行中档判定；首页 → 区（商户汇总）→ 点（单数）→ 标贴列表（一单一张） | 数据与对账清单口径一致 |
| **P2 · 完成作业** | 点完成、异常勾选、点级拍照上传、完成状态展示 | 进行中档可完成；其他档只读 |
| **P3 · 补拍与运营** | 历史档只读 + 补拍；管理员查看完成记录；打印/导出标贴（可选） | 昨日档补拍；运营追溯 |

**不建议一次完成**：P0 无任务也能上线后台录入；P1 无完成可先让配送员「看单」试运行；P2 再开放作业，风险可控。
