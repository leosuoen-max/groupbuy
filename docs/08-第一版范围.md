# 08 - 第一版范围

## 核心原则

**第一版要做到：让用户的太太能用起来。**

不追求完美，但要：
- 完整覆盖她每天的工作流（发菜单、收单、对账、出餐）
- 体验比群报数好（核心改进点要做到）
- 没有阻断性 bug

## 已知限制 / 第一版接受的妥协

### 1. iOS PWA 推送限制

- iOS 对 PWA 推送支持有限
- 第一版用 PWA 推送 + 商户主动刷新
- 不行就第二版做 WhatsApp Business API 推送

### 2. 顾客身份识别

- 游客订单依赖浏览器 localStorage
- 清缓存或换设备会丢失
- 提示游客："注册账号可永久保留订单记录"

### 3. 退款流程

- 平台不处理资金
- 顾客提交退款申请，商户线下处理
- 系统只记录状态

### 4. 商户实时通知

- 第一版仅靠 PWA 推送
- 商户也可主动刷新查看
- 不做 WhatsApp/邮件/SMS 兜底

## 估算工作量

| 模块 | 工作量 |
|---|---|
| 项目搭建 + Firebase 配置 | 1-2 天 |
| 顾客端核心 | 5-7 天 |
| 商户后台核心 | 7-10 天 |
| 注册登录 | 2-3 天 |
| 分享卡片 + PWA | 2-3 天 |
| 测试和修 bug | 3-5 天 |
| **合计** | **约 20-30 天** |

不是熟练开发者的速度，配合 Cursor 边做边学的速度，可能再多 30%-50%。

## 上线判断标准

第一版可以上线，需要满足：

- [ ] 用户的太太完整跑通"发布菜单 → 收单 → 核实付款 → 出餐 → 对账"全流程
- [ ] 至少 3 位顾客真实下单（朋友/邻居）
- [ ] 没有任何阻断性 bug（页面打不开、按钮没反应、数据丢失）
- [ ] 顾客端在微信、WhatsApp 中都能正常打开
- [ ] 商户端添加到桌面后能正常使用
- [ ] PWA 推送能收到（即使 iOS 偶尔失败也接受）
- [ ] 数据备份策略已实施（Firebase 自动备份）
- [ ] 有反馈入口收集用户问题
