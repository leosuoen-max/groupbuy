## 9.1 顾客完整下单流程

```

   ↓

   ↓

[加载页面]

   ├─ 浏览商品清单
   └─ 看到付款码
   ↓

   ├─ 1荤1素 ×1（点击 [+]）
   ├─ 米饭 ×1（点击 [+]）
   └─ 底部显示"已选 2 件，合计 RM 16.8"
   ↓
小明点击 [点此提交]
   ↓
跳转到下单页：填信息
   ├─ 姓名：李美玲
   ├─ 电话：6012-3456789
   ├─ 地址：B1-25-12, CASA Residenzi
   └─ 备注：（空）
   ↓
点击 [下一步]
   ↓

   ├─ 配送点 1 · A 座大堂
   ├─ 配送点 2 · B 座入口（小明选这个）
   ├─ 配送点 3 · 自取
   └─ 以上都不对（其他）
   ↓
点击 [提交订单]
   ↓
[系统处理]
├─ Firestore Transaction：
│  ├─ 检查库存（够）
│  ├─ 扣减库存
│  └─ 创建订单（编号 #L8）

   ↓

   ↓

   ↓
取餐时间到
   ↓
```

## 9.2 顾客加菜补付流程

```
小明已确认付款的订单 #L8
   ↓
小明发现忘记点饮料了
   ↓
小明打开"我的订单" → 找到 #L8 → 进入订单详情
   ↓
点击 [+ 加菜]
   ↓
跳转到选菜页面（已选商品保留）
├─ 1荤1素 ×1（已下单）
├─ 米饭 ×1（已下单）
└─ + 可乐 ×1（新加）
   ↓
点击 [确认加菜]
   ↓
[系统处理]
├─ 计算差额：RM 2.0（可乐价格）
├─ 更新订单：
│  ├─ items 增加可乐
│  ├─ totalAmount: 16.8 + 2.0 = 18.8
│  ├─ paidAmount: 16.8（不变）
│  ├─ pendingAmount: 2.0
│  └─ status: 'partial_paid'（待补付款）
└─ 添加状态历史记录
   ↓
显示提示："请补付 RM 2.0 并上传新的付款截图"
   ↓
小明扫付款码补付
   ↓
小明上传新截图
   ↓
[系统处理]
├─ 截图独立存储（不覆盖原截图）
├─ 订单状态：'pending'（待核实）
└─ 商户后台显示有新待核实
   ↓
商户确认 → 订单状态变 'confirmed'（全额付款）
```

## 9.4 状态流转图

### 9.4.1 项目状态

```
[创建草稿]
   ↓
draft（草稿）
   ↓ 点击发布
published（已发布）
   ↓ 截止时间到 / 商户手动截单
closed（已截止）
   ↓ 商户手动重新开放
published（已发布）
   ↓ ...
```

### 9.4.2 订单状态

```
[顾客提交订单]
   ↓
unpaid（待付款）
   ↓ 顾客上传截图
pending（待核实）
   ├─ 商户确认 → confirmed（已确认付款）
   ├─ 商户拒绝 → unpaid（退回待付款）
   └─ 顾客取消 → cancelled（已取消）
   ↓
confirmed（已确认付款）
   ├─ 顾客加菜 → partial_paid（待补付款）
   └─ 顾客申请退款 → refund_pending → refunded
   
partial_paid（待补付款）
   ↓ 顾客上传新截图
pending（待核实，针对差额部分）
   ↓ 商户确认
confirmed（全额已付）
```

### 9.4.3 截图状态

每张截图独立有状态：

```
[顾客上传]
   ↓
unverified（待核实）+ flag（绿/黄/红）
   ↓
   ├─ 商户确认 → verified
   └─ 商户拒绝 → rejected（带原因）
```

## 9.5 错误场景处理

### 9.5.1 网络断开

**顾客下单时网络断了：**
- Service Worker 缓存关键页面
- 显示"网络已断开，请检查后重试"
- 已加载的数据可继续浏览
- 重连后自动同步未提交的操作

**商户操作时网络断了：**
- 操作进入队列
- 显示"网络已断开，操作已保存"
- 重连后自动重试
- 重要操作（如确认付款）需要重连后用户手动重试

### 9.5.2 库存售罄

**顾客提交订单时商品已售罄：**
```
顾客点击 [提交订单]
   ↓
系统检查库存（事务）
   ↓
库存不足！
   ↓
显示弹窗：
├─ "抱歉，1荤1素 已售罄"
├─ "请重新选择其他商品"
└─ [返回选菜]
   ↓
返回选菜页，已售罄商品的 [+] 按钮变灰
```

### 9.5.3 截图上传失败

```
顾客选择图片
   ↓
[系统处理]
├─ 计算 MD5（前端）
├─ 压缩图片（可选）
└─ 上传到 Firebase Storage
   ↓
上传失败（网络/Storage 限额）
   ↓
显示错误：
├─ "上传失败，请重试"
├─ [重试]
└─ 不丢失已选的图片（保留在内存）
```

### 9.5.4 商户改菜单时已有订单

**情况 A：** 商户想改某商品的价格
- 系统检查：该商品有没有被下单
- 已被下单 → 提示"该商品已被下单，无法修改价格。建议下架后创建新商品"
- 没被下单 → 允许修改

**情况 B：** 商户想删除某商品
- 同上检查
- 已被下单 → 提示无法删除，可以下架
- 没被下单 → 允许删除

## 9.6 性能要求

| 场景 | 目标 |
|---|---|
| 顾客打开店铺首页 | < 2 秒（首次）/ < 1 秒（缓存）|
| 商品清单滚动 | 60fps，不卡顿 |
| 提交订单 | < 1 秒响应 |
| 上传付款截图（1MB）| < 5 秒 |
| 商户后台加载 | < 3 秒（含订单数据）|
| 实时同步延迟 | < 2 秒（Firestore 默认）|

## 9.7 关键边界情况

### 9.7.1 同名顾客

两个不同的顾客都叫"李美玲"：
- 用电话号码区分
- 商户后台显示 "李美玲 (6012-***6789)"

### 9.7.2 顾客电话填错

- 不影响下单（电话只是弱身份标识）
- 商户在订单详情可以编辑（修改顾客电话）

### 9.7.3 商户截止时间过早

- 假如商户设了"5 分钟后截止"
- 系统正常执行，到点关闭
- 商户可手动重新开放

### 9.7.4 顾客重复点击提交

- 提交按钮点击后立即变灰 disabled
- 防止双击产生两笔订单
- 提交完成或失败后才恢复
