# 饭团订单支付后端后续清单

本文记录本轮“先前端对齐”之后，后端仍需要补齐的接口、权限和状态能力。

## 订单查询接口

- 建议新增统一的饭团顾客订单查询接口，服务端按 `customerUserId`、`customerKey`、`wechatNotifyOAuthStateId` 归并订单，前端不再在多个页面拼多次 Firestore 查询。
- 接口应支持按项目过滤，用于项目内“我的订单”；也应支持全局查询，用于“我的饭团订单”。
- 服务端应在返回前处理限时优惠超时取消，保证列表和详情状态一致。

## 管理员订单列表

- 饭团后台订单量变大后，需要按 `channel + projectId + status/createdAt` 建索引，或增加项目/状态汇总文档，避免后台列表全量读。
- 待确认支付组、待付款支付组可以考虑由后端写入冗余摘要字段，方便管理员列表低成本筛选和排序。

## 支付状态机

- 需要明确饭团订单取消、退款、部分退款的状态流转，并定义库存回滚、钱包回滚和支付组确认撤销规则。
- 饭团钱包支付已经会自动确认支付组，后端后续应提供只读审计接口，方便管理员按订单、钱包账户、流水三方交叉核对。

## 钱包运营能力

- 需要补齐钱包账户停用/恢复、人工调账、充值申请撤销的数据模型和权限规则。
- 人工调账必须写钱包流水，并带操作人、原因、变更前后余额，避免只改账户余额。
- 充值申请建议增加“已读/待处理”或“风险标记”字段，用于后台运营优先处理异常截图。

## 权限与审计

- 饭团订单确认、免提交凭证、充值确认、充值驳回应统一校验饭团管理员权限。
- 所有资金相关写操作应写入 `statusHistory` 或专门审计日志，至少包含操作人、时间、动作、金额、关联订单/充值申请/流水 ID。
