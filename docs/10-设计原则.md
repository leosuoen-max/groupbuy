# 10 - 设计原则

## 10.1 核心设计哲学

### 10.1.1 简洁优于功能丰富

我们已经砍掉了很多功能（看 `08-第一版范围.md`），保持克制：
- 不做过度设计
- 不堆砌选项
- 让 80% 的用户用 20% 的功能就完成 100% 的事

### 10.1.2 真实场景优于设想

每个功能都要问：
- 辉姐每天会用这个吗？
- 顾客在群里看到链接，会立即明白怎么操作吗？
- 这个功能解决了什么真实问题？

## 10.2 UI 设计原则

### 10.2.1 移动优先

- 所有页面优先考虑手机屏幕（375-414px 宽度）
- PC 是附加场景，自适应即可
- 触摸区域不小于 44x44px

### 10.2.2 信息密度

- **顾客端：** 友好、温暖、不挤
  - 间距宽松，字号大
  - 图片大方，不缩略
  - 按钮明显，不羞涩
  
- **商户端：** 高效、紧凑、专业
  - 表格列表为主
  - 数字醒目
  - 操作快捷

### 10.2.3 视觉层次

```
重要 > 次要 > 辅助
颜色 > 字号 > 位置 > 字重
```

例：商品价格用红色加粗，备注用浅灰小字。

### 10.2.4 颜色语义

| 颜色 | 含义 |
|---|---|
| 🟢 绿色 | 成功、已确认、正常 |
| 🟡 黄色 | 注意、警告、待处理 |
| 🔴 红色 | 错误、拒绝、未付款 |
| 🟠 橙色 | 提醒、待补付款 |
| ⚪ 灰色 | 已截止、已取消、不可用 |
| 🔵 蓝色 | 信息提示（少用，避免与品牌色冲突）|

### 10.2.5 主题色

## 10.3 交互原则

### 10.3.1 反馈即时

任何操作都要有反馈：
- 点击按钮 → 立即变色或震动
- 提交表单 → 显示 loading
- 操作完成 → Toast 或弹窗提示
- 操作失败 → 明确说明原因

### 10.3.2 操作可逆

- 重要操作前确认（如删除订单）
- 删除截图需要二次确认
- 误操作可以撤销（如恢复刚删的商品）

### 10.3.3 默认值合理

- 注册时默认勾选"30 天内自动登录"
- 发布项目时默认主题色（继承上次）
- 配送点选择默认全选（继承上次）

### 10.3.4 防误触

- 危险按钮远离常用按钮
- 删除按钮颜色明显（红色）+ 二次确认
- 提交按钮防双击

## 10.4 文案原则

### 10.4.1 口语化

❌ "请提交您的订单信息"
✅ "立即下单"

❌ "操作失败，请重试"
✅ "提交失败，再试一下？"

### 10.4.2 友好

❌ "未登录用户无法使用此功能"
✅ "登录后就能用啦 → [登录]"

### 10.4.3 简洁

❌ "您选择的菜品已超过当前可购买数量"
✅ "已超过库存"

### 10.4.4 专业但不冰冷

❌ "本店辉煌厨艺，敬请期待"（过分）
✅ "今日新菜：小炒黄牛肉"（直接）

### 10.4.5 错误提示要给出路

❌ "提交失败"
✅ "提交失败，可能是网络问题。[重试]"

## 10.6 不要做的事（红线）

以下是核心红线，**绝对不要做**：

### ❌ 不要在顾客端塞任何第三方广告

群报数会塞广告，污染商户品牌。我们坚决不做。

### ❌ 不要在分享卡片上显示平台 Logo

我们的分享卡片只显示商户品牌（店名、Logo）。

### ❌ 不要做评分、评论、点赞功能

这是工具，不是社交平台。

### ❌ 不要强制注册

游客可以下单是核心。

### ❌ 不要默认把订单数据公开给其他顾客

订单可见性默认"仅自己可见"，保护隐私。

### ❌ 不要让顾客的付款截图被其他顾客看到

只有商户和管理员可见。

### ❌ 不要做自动判断付款是否成功

三色提示是辅助，最终人工判断。

### ❌ 不要做退款资金处理

我们不碰钱，让商户自己处理。

### ❌ 不要做"取餐二维码"

订单编号叫号已够，二维码核销是过度设计。

## 10.7 命名约定

### 10.7.1 文件命名

```
组件文件：PascalCase
  ✅ ProductCard.tsx
  ❌ product-card.tsx

工具/Hook 文件：camelCase
  ✅ useAuth.ts, formatDate.ts
  ❌ UseAuth.ts

样式文件：kebab-case
  ✅ globals.css

文档文件：kebab-case 或中文
  ✅ 03-顾客端功能.md
```

### 10.7.2 变量命名

```
// 业务名词用英文
const customerName = "李美玲"
const orderNumber = "L8"

// Boolean 用 is/has/can 前缀
const isLoading = false
const hasOrders = true
const canEdit = true

// 函数用动词开头
function calculateTotal() {}
function handleSubmit() {}
function fetchOrders() {}
```

### 10.7.3 CSS 类名

使用 Tailwind 工具类，不写自定义 CSS（除非必要）：

```jsx
✅ <div className="bg-red-500 text-white p-4 rounded">

❌ <div className="my-custom-button">
```

如果重复使用，提取成组件而不是 CSS 类。

## 10.8 国际化考虑

第一版只做中文，但代码要为多语言做好准备：

```typescript
// 不要硬编码中文
❌ <button>点此提交</button>

// 使用 i18n（即使第一版只有中文）
✅ <button>{t('order.submit')}</button>
```

预留以下语言：
- zh（中文，默认）
- en（英文）
- ms（马来文）

货币、日期格式：
- MYR（马来西亚林吉特，默认）
- 日期：DD/MM/YYYY

虽然不是首要目标，但做基础的：

- 表单元素有 label

- 颜色不是唯一信息载体（重要状态加文字说明）

## 10.10 组件设计原则

### 10.10.1 组件粒度

- 太大：维护困难，难复用
- 太小：碎片化，引用混乱
- 适中：单一职责，可测试

例：
```
✅ ProductCard.tsx（一个商品的展示）
✅ ProductList.tsx（多个 ProductCard 组合）
✅ ProductImage.tsx（如果图片处理逻辑复杂）

❌ AllInOneShopPage.tsx（什么都塞一起）
❌ Title.tsx, Subtitle.tsx, ParagraphText.tsx（过度拆分）
```

### 10.10.2 props 设计

```typescript
// 必填的放前面，选填放后面
interface ProductCardProps {
  product: Product;          // 必填
  onSelect: () => void;      // 必填
  showActions?: boolean;     // 选填
  className?: string;        // 选填
}
```

### 10.10.3 受控 vs 非受控

表单优先用受控组件（用 React Hook Form 管理）：

```jsx
✅ <input value={name} onChange={...} />
❌ <input defaultValue="..." />（非受控）
```

## 10.11 性能原则

### 10.11.1 该懒加载就懒加载

```typescript
// 路由懒加载
const Dashboard = lazy(() => import('./pages/Dashboard'));

// 图片懒加载
<img loading="lazy" src="..." />

// 组件懒加载（重型组件）
const ShareCardGenerator = lazy(() => import('./ShareCardGenerator'));
```

### 10.11.2 防抖和节流

```typescript
// 搜索输入：防抖
const debouncedSearch = useDebounce(searchTerm, 300);

// 滚动监听：节流
const throttledScroll = useThrottle(handleScroll, 100);
```

### 10.11.3 列表虚拟化

如果某个列表有可能超过 100 条，考虑虚拟列表：
- react-window 或 react-virtual

## 10.12 调试原则

### 10.12.1 控制台日志

开发环境多 log，生产环境少 log：

```typescript
if (import.meta.env.DEV) {
  console.log('[Order] created:', order);
}
```

### 10.12.2 错误处理

```typescript
try {
  await submitOrder(data);
} catch (error) {
  console.error('[OrderSubmit] failed:', error);
  toast.error('提交失败，再试一下？');
  // 上报到 Sentry（可选）
}
```

### 10.12.3 边界情况要测

每个功能完成后想想：
- 空状态：没数据时显示什么？
- 加载状态：等待时显示什么？
- 错误状态：失败时显示什么？
- 极限情况：100 条数据、1MB 图片、长文本截断
